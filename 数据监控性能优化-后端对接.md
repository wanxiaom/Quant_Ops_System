# 数据监控性能优化 — 后端对接说明（P0）

> 前端 P0 改造已完成，本文供与后端 Manager 对齐接口。  
> 关联文档：`数据监控对接文档.md`、类型定义 `src/types/domain.ts`。

---

## 1. 背景与问题

当前生产环境规模（估算）：

| 数据 | 规模 |
|------|------|
| 监控配置 `ddb_monitor` | **1000+** 条 |
| 快照 `ddb_monitor_snapshot` | **10000+** 条（约 1000 表 × 10 天） |

原前端一次拉全量，导致：

- 接口响应慢、JSON 体积大
- 浏览器解析与渲染压力大
- 两个 GET 并行时，慢接口拖垮整页加载

---

## 2. 前端 P0 已做改造（供联调参考）

| 项 | 行为 |
|----|------|
| 监控配置 Tab | **服务端分页**，每页 10 条，支持 `group_name` / `keyword` 筛选 |
| 监控概览 Tab | **默认选中分组「行情」**，必须带 `group_name` 才请求快照 |
| 概览矩阵行 | 按分组拉取启用中的监控项（`enabled=1`，`page_size=500`） |
| 页面加载 | **拆分请求**：概览与配置各自加载，不再 `Promise.all` 等最慢接口 |
| 配置 Tab | `lazy` 懒加载，切到该 Tab 才发配置列表请求 |
| 关键字搜索 | 概览侧 **300ms 防抖** 后再请求 |

---

## 3. P0 必须调整：`GET /api/ddb-monitors`

### 3.1 变更说明

由「返回全量数组」改为「**分页结构**」。前端已按新结构实现，并兼容旧版全量数组（过渡期）。

### 3.2 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `page` | int | 否 | 页码，从 **1** 开始，默认 `1` |
| `page_size` | int | 否 | 每页条数，配置 Tab 传 `10`；概览 Tab 传 `500` |
| `group_name` | string | 否 | 业务分组精确匹配：行情 / 因子 / 财务 / 基础数据 / 其他 |
| `keyword` | string | 否 | 库名、表名模糊搜索 |
| `enabled` | int | 否 | `1` 仅启用，`0` 仅禁用；不传则不过滤 |

**示例 — 监控配置 Tab**

```http
GET /api/ddb-monitors?page=1&page_size=10&group_name=行情&keyword=daily
```

**示例 — 监控概览 Tab（拉取矩阵行）**

```http
GET /api/ddb-monitors?group_name=行情&enabled=1&page=1&page_size=500
```

### 3.3 响应结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "monitor_id": "mon_market_daily_bar",
        "database": "dfs://market",
        "table_name": "daily_bar",
        "date_column": "trade_date",
        "date_format": "YYYY-MM-DD",
        "group_name": "行情",
        "related_task_ids": ["task_market_stock_daily"],
        "enabled": 1,
        "description": "A 股日频行情主表",
        "created_at": "2026-06-01T10:00:00+08:00",
        "updated_at": "2026-06-20T10:00:00+08:00"
      }
    ],
    "total": 1024,
    "page": 1,
    "page_size": 10
  }
}
```

### 3.4 后端实现建议

1. **先 `WHERE` 过滤，再 `COUNT(*)` 得 `total`，最后 `LIMIT/OFFSET`**
2. 建议索引：`group_name`、`enabled`，表名/库名搜索可用 `LIKE` + 合适索引
3. `page_size` 建议上限 **500**（概览单次拉取一个分组）
4. 开启 **gzip** 压缩

### 3.5 兼容性

前端 `ddbMonitorApi.list()` 若收到旧版 `data: DdbMonitor[]`，会自动包装为分页结构。  
**建议后端尽快切到新结构**，旧格式仅作过渡。

## 4. P0 必须落实：`GET /api/ddb-monitors/snapshots`

接口路径不变，但请确保以下查询参数**在服务端生效**（前端已传）：

| 参数 | 前端行为 | 后端要求 |
|------|----------|----------|
| `date_from` | 最近 10 个工作日最早一天 | **必填建议**，避免全表扫描 |
| `date_to` | 最近 10 个工作日最晚一天 | **必填建议** |
| `group_name` | 概览 Tab **必传**（默认「行情」） | **必须支持**，显著缩小结果集 |
| `keyword` | 库名/表名搜索（可选） | 建议支持 |

**示例**

```http
GET /api/ddb-monitors/snapshots?date_from=2026-06-27&date_to=2026-07-10&group_name=行情
```

### 4.1 数据量预期（优化后）

| 场景 | 优化前 | 优化后（单分组） |
|------|--------|------------------|
| 配置列表 | ~1000 条/次 | **10 条/次** |
| 快照列表 | ~10000 条/次 | **~分组表数 × 10 天**（如 200×10=2000） |

### 4.2 后端实现建议

1. 索引：`(group_name, date)` 或 `(monitor_id, date)`
2. 只查 `enabled=1` 的监控项对应快照（若业务允许）
3. 响应字段保持与 `DdbMonitorSnapshot` 一致，见 `数据监控对接文档.md`

## 6. P1 规划（本期不做，供后端预留）

当单分组仍超过 500 表时，建议新增矩阵专用接口：

```http
GET /api/ddb-monitors/matrix?group_name=行情&date_from=...&date_to=...&page=1&page_size=50
```

响应包含：

- `dates[]`：列头日期
- `rows[]`：每行含 `monitor_id`、库表信息、`cells[]`（`date` / `row_count` / `status`）
- `total`：行总数
- `summary`：未统计/正常/数据异常/统计失败计数

可显著减少冗余字段与前端拼装成本。详见前期性能评估讨论。
