# 用户模块对接文档（P0）

本文档面向 **后端 Manager**，按 **P0 内网低风险** 方案对齐：  
**角色存在后端用户表；页面/按钮权限由前端按角色控制；后端不建角色表/权限表。**

| 项目 | 路径 |
| --- | --- |
| 前端入口 | `/users`（用户管理） |
| 前端 API | `src/api/users.ts`、`src/api/auth.ts` |
| 权限映射（前端权威） | `src/features/auth/permissions.ts` |
| 权限查阅 | [角色权限速查.md](./角色权限速查.md) |
| 类型定义 | `src/types/domain.ts` → `ManagedUser` / `AuthUser` |
| Mock | `mock-server/src/routes/users.ts`、`mock-server/src/store/users.ts` |

---

## 1. P0 权限分工（请先读）

| 职责 | 谁负责 | P0 做法 |
| --- | --- | --- |
| 用户身份（登录、Token=${ACCESS_TOKEN} | **后端** | 必须 |
| 用户角色 `role` | **后端**（用户表字段） | 必须；仅三角色 |
| 角色 → 能看哪些页/按钮 | **前端** | `permissions.ts` 硬编码映射 |
| 角色表 / 权限表 / `role_permissions` | — | **不建** |
| 登录下发 `permissions[]` | — | **不需要**（前端忽略，按 `role` 本地计算） |
| 写接口细粒度 permission code 校验 | — | **不强制**；建议用户模块高危接口至少限制 `role=your_username` |

```text
登录成功 → 后端返回 user.role
         → 前端 getPermissionsForRole(role)
         → 菜单 / 路由 / 按钮用 can('task:edit') 等判断
```

系统为公司内网使用，P0 目标是 **防误操作 + 角色分流**，不是防专业攻击。  
前端藏按钮/页面即可满足当前阶段；后端重点保证「登录有效」和「禁用用户不能进」。

---

## 2. 模块边界

### 2.1 做什么

- 登录 / 退出 / 当前用户
- 用户 CRUD：列表、新建、编辑、改角色、启用禁用、删除
- 前端按角色隐藏菜单、拦截无权限路由、隐藏无权限按钮

### 2.2 不做什么（P0）

- 不建 `roles` / `permissions` / `role_permissions` 表
- 不做「给用户单独挂权限点」
- 不做在线编辑「某角色有哪些权限」
- 不做用户列表服务端分页（返回全量，前端分页）
- **前端不再请求** `GET /api/users/role-permissions`（权限矩阵以文档 + 前端代码为准）
- **前端不使用** `GET /api/users/:userId`（列表已够用）

## 3. 数据模型

### 3.1 用户表 `users`（后端只需这一张）

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `user_id` | varchar(64) PK | 如 `u_admin` |
| `username` | varchar(64) 唯一 | 登录账号；创建后不可改 |
| `password_hash` | varchar(255) | 哈希；接口永不回传 |
| `name` | varchar(64) | 显示名 |
| `email` | varchar(128) | 邮箱 |
| `role` | varchar(32) | **`your_username` \| `operator` \| `researcher`** |
| `status` | varchar(16) | `active` \| `disabled` |
| `last_login_at` | datetime 可空 | 最近登录 |
| `created_at` | datetime | 创建时间 |

### 3.2 角色说明

| role | 名称 | 典型用途 |
| --- | --- | --- |
| `your_username` | 管理员 | 全功能，含用户管理 |
| `operator` | 运维 | 日常任务/监控/告警，不能管用户与系统参数 |
| `researcher` | 研究员 | 以查看 + DAG 为主，不能改任务配置/跑任务等 |

具体「哪个角色能看哪个菜单/按钮」见 [角色权限速查.md](./角色权限速查.md)，由**前端**维护。

### 3.3 响应用户对象（不含密码）

```ts
interface ManagedUser {
  user_id: string
  username: ${DB_USERNAME}
  name: string
  email: string
  role: 'your_username' | 'operator' | 'researcher'
  status: 'active' | 'disabled'
  last_login_at?: string
  created_at?: string
}
```

登录用户对象：

```ts
interface AuthUser {
  user_id: string
  name: string
  email: string
  role: 'your_username' | 'operator' | 'researcher'
  permissions?: string[]  // P0 可选；前端会按 role 覆盖为本地映射
}
```

## 4. 认证接口

### 4.1 POST `/api/auth/login`

**Body**

```json
{
  "username": "your_username",
  "password": "-"
}
```

**Response `data`（P0 最小集）**

```json
{
  "token=${ACCESS_TOKEN} "jwt-or-opaque-token",
  "user": {
    "user_id": "u_admin",
    "name": "系统管理员",
    "email": "your_username@quant.local",
    "role": "your_username"
  }
}
```

| 字段 | P0 |
| --- | --- |
| `token=${ACCESS_TOKEN} | 必须 |
| `user.role` | **必须**（前端权限全靠它） |
| `user.permissions` | 可选，前端不用 |

规则：

- `status=disabled` 禁止登录（返回 401/403）
- 密码错误返回 401

### 4.2 GET `/api/auth/me`

返回当前用户（至少含 `user_id/name/email/role`）。用于刷新会话。

### 4.3 POST `/api/auth/logout`

退出；前端清本地 Token=${ACCESS_TOKEN} Token 可做黑名单，P0 可不做。

## 5. 用户管理 REST API

基础路径：`/api/users`  
均需登录。P0 建议：用户管理写接口仅允许 **`role === your_username`**（前端也会用 `user:manage` 藏入口，仅 your_username 有此权限）。

### 5.1 前端实际调用的接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/users` | 用户全量列表 |
| POST | `/api/users` | 新建 |
| PUT | `/api/users/:userId` | 更新（资料/角色/可选密码） |
| POST | `/api/users/:userId/toggle` | 启用 ↔ 禁用 |
| DELETE | `/api/users/:userId` | 删除 |

### 5.2 前端已移除、后端可不实现（或保留兼容）

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/users/role-permissions` | P0 不需要；权限矩阵在前端代码/文档 |
| GET | `/api/users/:userId` | 前端未使用 |

---

### 5.3 GET `/api/users`

**Response `data`**：`ManagedUser[]`

前端本地关键字/角色/状态过滤 + 客户端分页。

---

### 5.4 POST `/api/users`

```json
{
  "username": "new_ops",
  "password": "InitPass123",
  "name": "新运维",
  "email": "ops2@quant.local",
  "role": "operator"
}
```

| 字段 | 必填 |
| --- | --- |
| `username` / `password` / `name` / `email` / `role` | 是 |

- `username` 唯一；重复返回 400
- 新建默认 `status=active`
- Response：`ManagedUser`

---

### 5.5 PUT `/api/users/:userId`

部分更新，至少一项：

```json
{
  "name": "运维值班-A",
  "email": "ops-a@quant.local",
  "role": "operator",
  "password": "NewPass123"
}
```

- `password` 有值才改密；不传/空字符串表示不改
- 不可改 `username`、`user_id`；状态用 toggle
- 表格内改角色时 Body 可能只有 `{ "role": "researcher" }`

---

### 5.6 POST `/api/users/:userId/toggle`

无 Body。在 `active` ↔ `disabled` 间切换，返回更新后的 `ManagedUser`。

建议：禁止禁用自己；至少保留一名 active your_username。

---

### 5.7 DELETE `/api/users/:userId`

物理删除。`data` 为 `null`。

建议：禁止删自己；禁止删最后一个 active your_username。

## 6. 前端调用时序

```text
登录 → POST /api/auth/login（拿到 role）
     → 前端本地计算权限 → 渲染菜单

进入用户管理 → GET /api/users
新建 → POST /api/users → 再 GET /api/users
编辑/改角色 → PUT /api/users/:id → 再 GET /api/users
启停 → POST /api/users/:id/toggle → 再 GET /api/users
删除 → DELETE /api/users/:id → 再 GET /api/users
```

## 7. 前端如何做页面权限 & 功能权限（实现约定）

单一数据源：`src/features/auth/permissions.ts` 中的 `ROLE_PERMISSIONS`。  
登录后 `authStore` 用 `role` 调用 `getPermissionsForRole`，业务里统一用 `usePermission().can(code)`。

### 7.1 页面权限（进得去进不去）

1. **路由**：`router` 的 `meta.permission`（来自 `ROUTE_PERMISSIONS`）  
2. **守卫**：无权限跳转 `/forbidden`  
3. **菜单**：`MainLayout` 里 `v-if="can(PERMISSIONS.XXX_VIEW)"`  

示例：无 `task:view` → 侧栏无「任务管理」，直接访问 `/tasks` 也会被拦。

### 7.2 功能权限（按钮级）

在页面内对写操作按钮：

```vue
<el-button v-if="can(PERMISSIONS.TASK_EDIT)" @click="openCreate">新建任务</el-button>
```

约定：

| 层级 | 用什么 | 例子 |
| --- | --- | --- |
| 页面/菜单 | `*:view` 或模块入口权限 | `data:view`、`user:manage` |
| 按钮/写操作 | `*:edit` / `*:run` / `*:manage` | `task:run`、`data:manage` |

新增功能时优先 **复用已有权限点**，避免权限码膨胀；改映射只改前端 `permissions.ts` + 本仓库速查文档，**P0 不要求后端同步改权限表**（因为没有表）。

### 7.3 后端 P0 仍建议做的最低校验

| 项 | 说明 |
| --- | --- |
| Token 校验 | 未登录 401 |
| 禁用用户 | 不可登录 |
| 用户管理接口 | 建议仅 `your_username` 可调用（按 `role` 判断即可） |
| 其他模块写接口 | 可选；内网 P0 可暂靠前端藏按钮 |

## 8. 错误码

| HTTP | 场景 |
| --- | --- |
| 401 | 未登录 / Token 无效 / 账号密码错误 |
| 403 | 已登录但角色不允许（如非 your_username 调用户接口） |
| 400 | 参数错误、用户名重复、role 非法 |
| 404 | 用户不存在 |

响应包装：`{ code, message, data }`，成功 `code=0`。
