# 日志中心分页对接文档（P0）

本文档面向 **后端 Manager**，说明日志列表从「前端全量拉取再筛选」改为 **服务端分页 + 筛选** 的接口约定。  
对照实现可参考数据监控库表配置：`GET /api/ddb-monitors?page=&page_size=&group_id=&keyword=`。

| 项目 | 路径 |
| --- | --- |
| 前端页面 | `/logs`（`src/views/LogsView.vue`） |
| 现有（未分页）API 封装 | `src/api/logs.ts` → `logApi.search` |
| 详情 / 下载（不变） | `GET /api/logs/:execId`、`GET /api/logs/:execId/download` |
| 类型 | `src/types/domain.ts` → `LogEntry` / `LogSearchParams` |

---

## 0. 现状说明（重要）

| 项 | 现状 |
| --- | --- |
| 列表数据来源 | **不是** `GET /api/logs/search`，而是拉全量 `GET /api/executions` + 任务列表，在浏览器拼成日志行 |
| 筛选 | **纯前端**过滤（任务名、节点、状态、日期） |
| 分页 | **未实现服务端分页**；列表直接渲染过滤后的全部结果，也未接 `ListPagination` |
| `logApi.search` | 代码里已有，但 **日志页当前未使用**；且仍按「返回全量数组」设计 |

因此：**后端只上分页不够**。需要：

1. 后端实现分页版 `GET /api/logs/search`
2. 前端改为调用该接口，接上分页组件，去掉「全量 executions 本地筛」主路径

---

## 1. 目标行为

与库表配置一致：

```text
筛选条件变化 / 翻页 / 改每页条数
  → GET /api/logs/search?page=&page_size=&...筛选参数
  → 展示当前页 items，分页器 total 用后端 total
```

- 改筛选：重置到 `page=1` 再请求  
- 任务名输入：建议 **300ms 防抖** 后再请求（避免每个字母打一次）  
- 详情、下载仍按 `exec_id` 调现有接口，无分页需求  

---

## 2. 接口变更：`GET /api/logs/search`

### 2.1 Query 参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `page` | int | 否 | 页码，从 **1** 开始，默认 `1` |
| `page_size` | int | 否 | 每页条数，默认 `20`；建议上限 **100** |
| `keyword` | string | 否 | 模糊匹配 **任务名或 task_id**（对应前端「任务名称」搜索框） |
| `agent_id` | string | 否 | 执行节点精确匹配（`node_id`） |
| `status` | string | 否 | `waiting` \| `running` \| `success` \| `failed` \| `timeout` |
| `date_from` | string | 否 | 开始日期 `YYYY-MM-DD`（含当日） |
| `date_to` | string | 否 | 结束日期 `YYYY-MM-DD`（含当日） |

> **日期参数建议**用 `date_from` / `date_to`，不要用数组 `date_range`（query 传数组各端实现不一致）。  
> 前端日期范围选择器会拆成这两个字段。  
> 旧契约中的 `task_name`、`level`、`date_range`：**P0 可不实现**；有 `keyword` 即可覆盖任务名搜索。

**示例**

```http
GET /api/logs/search?page=1&page_size=20&keyword=行情&agent_id=node-01&status=failed&date_from=2026-07-01&date_to=2026-07-14
```

### 2.2 响应结构（与监控分页对齐）

由「全量数组」改为分页对象：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "log_id": "exec_1001",
        "exec_id": "exec_1001",
        "task_id": "task_market_stock_daily",
        "task_name": "股票行情日频同步",
        "agent_id": "node-01",
        "level": "INFO",
        "time": "2026-07-14 09:15:00",
        "status": "success"
      }
    ],
    "total": 1286,
    "page": 1,
    "page_size": 20
  }
}
```

```ts
interface PaginatedResult<T> {
  items: T[]
  total: number
  page: number
  page_size: number
}

interface LogEntry {
  log_id: string      // 可用 exec_id
  exec_id: string
  task_id: string
  task_name: string
  agent_id: string
  level: 'INFO' | 'WARN' | 'ERROR'
  time: string        // 建议用 end_time，无则 start_time；展示用
  status: 'waiting' | 'running' | 'success' | 'failed' | 'timeout'
}
```

列表项**不要**带大段 `stdout`/`stderr`/`content`，详情走 `GET /api/logs/:execId`。

### 2.3 排序

默认按执行时间 **倒序**（最近在前）：优先 `end_time`，否则 `start_time`。

### 2.4 后端实现建议

1. 以执行记录表（或等价存储）为主表联任务名  
2. `WHERE` 应用筛选 → `COUNT(*)` 得 `total` → `ORDER BY ... DESC LIMIT/OFFSET`  
3. 建议索引：`start_time` / `end_time`、`status`、`node_id`、`task_id`  
4. `keyword`：`task_name LIKE` 或 `task_id LIKE`（注意前缀模糊性能）

## 3. 顶部统计条（Summary）

当前前端在**本地过滤结果**上统计：总数 / 成功 / 运行中 / 失败+超时。

服务端分页后本地算不准，P0 任选其一：

| 方案 | 说明 | 推荐 |
| --- | --- | --- |
| A. 列表响应附带 `summary` | 与当前筛选条件一致的聚合计数 | **推荐** |
| B. 独立接口 | `GET /api/logs/summary?...同筛选参数`（不含 page） | 也可 |
| C. 暂弱化 | 只展示 `total`，去掉分状态 chip | 最快上线 |

**方案 A 示例**（扩展 `data`）

```json
{
  "items": [],
  "total": 1286,
  "page": 1,
  "page_size": 20,
  "summary": {
    "total": 1286,
    "success": 1100,
    "running": 12,
    "errors": 174
  }
}
```

`errors` = `failed + timeout`（与现前端一致）。  
`summary` 按**当前筛选**聚合，不受 `page` 影响。

## 4. 不变的接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/api/logs/:execId` | 详情，含 `content` |
| GET | `/api/logs/:execId/download` | 纯文本下载 |

路由注册顺序：`/search`（及若有 `/summary`）必须在 `/:execId` 之前。

节点下拉仍可用现有 `GET /api/agents`（或前端缓存），不必打进 search。

## 6. 与旧契约的差异

| 旧（`API接口契约.md`） | 本 P0 |
| --- | --- |
| `data: LogEntry[]` | `data: { items, total, page, page_size }` |
| `date_range` 数组 | `date_from` + `date_to` |
| 可选 `task_name` / `level` | P0 用 `keyword` 即可；`level` 暂不筛 |
| 页面用 `/api/executions` 全量 | 列表改走 `/api/logs/search` |

实现本对接后，请同步改 `API接口契约.md` 第 8 节，或以本文为准。
