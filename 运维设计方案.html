核心通信链路：

  • 前端 ↔ 管理端：HTTP REST + WebSocket 实时推送任务状态和日志
  • 管理端 ↔ Agent：Agent 主动发起 HTTP 长轮询或 WebSocket 连接，拉取任务指令并回传执行结果
  • 管理端 ↔ 数据库：通过 ormpp 连接池访问 MySQL，Redis 用于任务队列和状态缓存
  • Agent ↔ 现有脚本：通过 subprocess 调用，完全不修改原有 Python/Shell 逻辑

二、分布式更新节点设计（重点）

每个分布式节点上运行一个 轻量 Agent，负责接收管理端下发的指令、执行 ETL 脚本、上报执行结果和心跳。

2.1 Linux Agent（Python 实现）

部署环境： Ubuntu/CentOS 服务器，已部署原有 quant_data 项目及 crontab 任务。

功能模块：

模块说明
心跳上报上报 CPU、内存、正在运行的任务列表
任务拉取长轮询或 WebSocket 连接，接收管理端指令
脚本执行器通过 subprocess 隔离执行 Python/Shell 脚本
日志采集实时采集 stdout/stderr，压缩上传
并发控制限制同时运行的子进程数量（可选）

与现有系统的集成细节：

现有资产集成方式示例
screen -r quant_data 会话Agent 不再依赖 screen，直接拉起脚本Popen(["python", "/home/.../market/download.py"])
Crontab 定时任务停用 crontab，改由管理端下发调度策略原 30 16 * * * → 管理端配置 cron
本地缓存 (Pickle/Parquet)完全保留，Agent 不干预脚本内 pickle.load 不变
连接重试、差分更新保留在脚本内部脚本返回 0 成功，非 0 失败

核心代码骨架（Python）：

import subprocess, requests, time, json, psutil, socket
from pathlib import Path

class LinuxAgent:
    def __init__(self, manager_url, node_id):
        self.manager = manager_url.rstrip('/')
        self.node_id = node_id
        self.session = requests.Session()
        
    def _get_heartbeat_payload(self):
        return {
            "node_id": self.node_id,
            "ip": socket.gethostbyname(socket.gethostname()),
            "os_type": "linux",
            "last_heartbeat": time.time(),
            "cpu_load": psutil.cpu_percent(interval=1),
            "mem_usage": psutil.virtual_memory().percent,
            "running_tasks": self._get_running_subprocesses()
        }
    
    def _heartbeat_loop(self, interval=30):
        while True:
            try:
                resp = self.session.post(f"{self.manager}/api/agents/heartbeat",
                                         json=self._get_heartbeat_payload(), timeout=10)
                if resp.status_code != 200:
                    print("Heartbeat failed")
            except Exception as e:
                print(f"Heartbeat error: {e}")
            time.sleep(interval)
    
    def _execute_task(self, task):
        script_path = Path(task["script_path"])
        if not script_path.exists():
            return {"exit_code": -1, "stderr": f"Script {script_path} not found"}
        cmd = [task.get("interpreter", "python3"), str(script_path)]
        for k, v in task.get("params", {}).items():
            cmd.append(f"--{k}")
            if v is not None:
                cmd.append(str(v))
        proc = subprocess.Popen(cmd, stdout=subprocess.PIPE, stderr=subprocess.PIPE, text=True)
        try:
            stdout, stderr = proc.communicate(timeout=task.get("timeout", 3600))
        except subprocess.TimeoutExpired:
            proc.kill()
            stdout, stderr = proc.communicate()
            return {"exit_code": -2, "stderr": "Timeout expired", "stdout": stdout}
        return {
            "exit_code": proc.returncode,
            "stdout": stdout[-10000:],
            "stderr": stderr[-10000:]
        }
    
    def _task_pull_loop(self):
        while True:
            try:
                resp = self.session.post(f"{self.manager}/api/tasks/poll",
                                         json={"node_id": self.node_id, "timeout": 60}, timeout=70)
                if resp.status_code == 200:
                    data = resp.json()
                    if data.get("task"):
                        task = data["task"]
                        result = self._execute_task(task)
                        self.session.post(f"{self.manager}/api/tasks/callback",
                                          json={"exec_id": task["exec_id"], **result})
            except Exception as e:
                print(f"Poll error: {e}")
            time.sleep(1)
    
    def run(self):
        import threading
        threading.Thread(target=self._heartbeat_loop, daemon=True).start()
        self._task_pull_loop()

部署方式： 作为 systemd 服务，开机自启，自动重启。

2.2 Windows Agent（Python 实现）

部署环境： Windows 10/Server 办公机，必须保留 Wind 图形客户端。

特殊挑战： Wind API (WindPy) 需要有 GUI 的 Windows 环境；原有 main.py 使用 concurrent.futures 并行执行任务字典；需要兼容现有任务计划程序的双轨运行。

设计思路： Agent 作为包装器，通过进程调用现有脚本。

集成策略：

场景做法
管理端下发“更新ST股票”python st_stock_list.py --date 2026-06-03
管理端下发“全量更新”python main.py（Agent 仅监控进程）
增量补数传递 --start_date 参数给 download_data_from_wind_wsd.py

核心代码片段：

def run_windows_task(task):
    script = task["script_path"]
    cmd = ["python", script]
    for k, v in task.get("params", {}).items():
        cmd.extend([f"--{k}", str(v)])
    proc = subprocess.Popen(cmd, shell=True, stdout=subprocess.PIPE, stderr=subprocess.PIPE)
    stdout, stderr = proc.communicate(timeout=task.get("timeout", 3600))
    return {"exit_code": proc.returncode, "stdout": stdout, "stderr": stderr}

部署方式： 打包为 Windows 服务（PyInstaller + sc create）或后台进程。

2.3 DolphinDB 节点处理

方案一（推荐）： 不单独部署 Agent，将 DolphinDB 的库内计算脚本（如 all_databases.py）视为普通 Python 任务，调度到一台可访问 DolphinDB 的 Linux 节点执行。该节点安装 dolphindb Python 包即可。

方案二（利用 DolphinDB 定时作业）： DolphinDB 内置 HTTP 接口，管理端可通过 POST 触发脚本。
示例：curl -X POST http://dolphin-ip:8848/api/run -d '{"script": "runAllDatabases()"}'

实际建议： 采用方案一，保持 Agent 类型统一，降低复杂度。

2.4 Agent 通用设计原则

设计点要求
轻量内存占用 < 50MB
无状态重启后重新注册，任务重试由管理端负责
容错断连时不缓存任务，重连后重新拉取
安全只执行管理端下发的任务，不暴露额外接口
日志实时回传 stdout/stderr，管理端最终写入 ES

3.1 libhv HTTP/WebSocket 服务

基础服务代码：

#include "hv/HttpServer.h"
#include "hv/WebSocketServer.h"
#include "hv/hlog.h"

class ApiResponse {
public:
    static void Success(HttpResponse* resp, const Json& data) {
        Json json;
        json["code"] = 0;
        json["message"] = "success";
        json["data"] = data;
        resp->Json(json);
    }
    static void Error(HttpResponse* resp, int code, const std::string& msg) {
        Json json;
        json["code"] = code;
        json["message"] = msg;
        resp->Json(json);
    }
};

void TaskApi::incrUpdate(HttpRequest* req, HttpResponse* resp) {
    Json body = req->json();
    std::string taskId = body.value("task_id", "");
    std::string startDate = body.value("start_date", "");
    // 参数校验与调用
    bool ok = taskService->triggerTask(taskId, startDate, body.value("symbols", std::vector<std::string>()));
    if (ok) ApiResponse::Success(resp, {{"msg", "task triggered"}});
    else ApiResponse::Error(resp, 500, "trigger failed");
}

int main() {
    HttpService router;
    router.GET("/api/tasks", TaskApi::listTasks);
    router.POST("/api/tasks/{task_id}/incr", TaskApi::incrUpdate);
    router.STATIC("/", "./web_dist");
    
    http_server_t server;
    server.port = 8080;
    server.service = &router;
    http_server_run(&server, 0);
    return 0;
}

六、技术选型总结

组件技术理由
管理端C++17 + libhv + ormpp高性能（QPS 6W+）、低资源占用
前端Vue 3 + WebSocket实时状态监控、组件化开发
Agent (Linux/Win)Python + subprocess复用现有 ETL 脚本，零侵入
数据库MySQL + Redis持久化 + 实时队列/缓存
部署Docker Compose环境一致、快速迭代
