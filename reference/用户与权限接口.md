# 用户与权限接口

本文档定义 Quant Ops 前端的 RBAC 权限模型，供 **mock-server 实现参考** 与 **后端 Manager 对齐**。前端权限常量见 `src/features/auth/permissions.ts`，mock-server 镜像见 `mock-server/src/lib/permissions.ts`。

## 设计原则

1. **认证（Authentication）**：确认用户身份，签发 Token=${ACCESS_TOKEN}
2. **授权（Authorization）**：基于角色（Role）映射权限点（Permission Code），前后端共用同一套 code。
3. **后端强校验**：前端隐藏按钮/菜单仅为体验层，写操作接口必须在服务端校验权限。
4. **研究员账号**：`researcher` 可查看业务数据，拥有 DAG 全流程权限，不可执行任务或编辑任务/Agent 配置。

### 权限模型建议（给后端）

当前采用 **RBAC（基于角色的访问控制）**，权限点与角色的映射在代码中维护（`permissions.ts`），前后端共用同一套 permission code。

| 层级 | 当前做法 | 建议 |
|------|----------|------|
| 用户 | 用户表存 `role` 字段 | 后端只需 `users` 表：`user_id, username, password_hash, name, email, role, status` |
| 角色 | 固定三角色：`your_username` / `operator` / `researcher` | 无需 `roles` 表（P0 阶段） |
| 权限 | 代码硬编码「角色 → 权限点」映射 | **无需** `permissions` / `role_permissions` 表（P0 阶段） |
| 登录响应 | 根据 `role` 计算 `permissions[]` 返回 | 后端中间件按同样映射校验写接口 |

**用户管理页只支持「给用户分配角色」**，不支持：
- 单独编辑某个用户的权限点（无 per-user permission）
- 在 UI 中动态修改角色的权限集合（无 role editor）

若未来需要「自定义角色」或「给某人额外开权限」，再引入 `roles` + `role_permissions` 表，或 `user_extra_permissions` 表。当前阶段保持简单即可。

## 角色定义

| role | 名称 | 说明 |
|------|------|------|
| `your_username` | 管理员 | 全量权限 |
| `operator` | 运维 | 日常运维：任务/流程/告警，不可管用户与系统参数 |
| `researcher` | 研究员 | 查看 + DAG 编排/执行，不可改任务/Agent/告警/系统参数 |

## 认证接口

### POST `/api/auth/login`

登录并返回 Token 与用户信息（含权限列表）。

**请求体**

```json
{
  "username": "your_username",
  "password": "admin123"
}
```

**响应 `data`**

```ts
interface LoginResponse {
  token=${ACCESS_TOKEN}
  user: AuthUser
}

interface AuthUser {
  user_id: string
  name: string
  email: string
  role: 'your_username' | 'operator' | 'researcher'
  permissions: string[]   // 当前用户有效权限 code 列表
}
```

**Mock 测试账号**

| 用户名 | 密码 | 角色 | Token |
|--------|------|------|-------|
| your_username | admin123 | your_username | mock-token=${ACCESS_TOKEN} |
| operator | ops123 | operator | mock-token=${ACCESS_TOKEN} |
| researcher | researcher123 | researcher | mock-token=${ACCESS_TOKEN} |

开发 Token `quant_ops_secret_2026` 在 mock-server 中视为 your_username。

### GET `/api/auth/me`

根据 `Authorization: Bearer <token>` 返回当前用户，结构同 `AuthUser`。

### GET `/api/auth/permissions`

返回当前登录用户的角色与权限（可选，便于调试）。

**响应 `data`**

```json
{
  "role": "operator",
  "permissions": ["dashboard:view", "task:view", "task:edit", "..."]
}
```

### POST `/api/auth/logout`

登出（mock 为无状态，前端清除 Token 即可）。

## 用户管理接口

用户管理为独立模块，路径前缀 `/api/users`。写操作均需 `user:manage` 权限。

### GET `/api/users`

返回用户列表（不含密码）。

```ts
interface ManagedUser {
  user_id: string
  username: ${DB_USERNAME}
  name: string
  email: string
  role: UserRole
  status: 'active' | 'disabled'
  last_login_at?: string
  created_at?: string
}
```

### GET `/api/users/role-permissions`

返回角色权限矩阵：

```ts
interface RolePermission {
  role: UserRole
  permissions: string[]   // 权限 code 列表
}
```

### POST `/api/users`

创建用户。**需要权限**：`user:manage`

```json
{
  "username": "new_ops",
  "password": "opsyour_password",
  "name": "新运维",
  "email": "new_ops@quant.local",
  "role": "operator"
}
```

### PUT `/api/users/:userId`

更新用户。**需要权限**：`user:manage`

```json
{
  "name": "新运维",
  "email": "new_ops@quant.local",
  "role": "operator",
  "password": "new_password"
}
```

`password` 可选，留空或不传表示不修改密码。

### POST `/api/users/:userId/toggle`

启用/禁用用户。**需要权限**：`user:manage`

### DELETE `/api/users/:userId`

删除用户。**需要权限**：`user:manage`

## 系统设置接口

### GET `/api/settings/overview`

返回告警规则、通知渠道、系统参数快照（不再包含用户列表）。

## 业务接口权限要求（写操作）

后端实现时，建议对以下接口校验权限（mock-server 已按此实现）：

| 接口 | 所需权限 |
|------|----------|
| `POST/PUT/DELETE /api/tasks*` | `task:edit` |
| `POST /api/tasks/:id/run` | `task:run` |
| `POST/PUT/DELETE /api/pipelines*` | `pipeline:edit` |
| `POST /api/pipelines/:id/publish` | `pipeline:edit` |
| `POST /api/agents` | `agent:manage` |
| `POST /api/agents/:id/offline` | `agent:manage` |
| `GET/POST/PUT/DELETE /api/users*` | `user:manage` |
| `PUT/POST /api/settings/alert-rules*` | `alert:manage` |
| `PUT/POST /api/settings/notification-channels*` | `alert:manage` |
| `PUT/POST /api/settings/system-parameters*` | `system:manage` |

读接口（GET）默认要求已登录；细粒度可读权限可按需追加。

## 错误码

| HTTP | code | 场景 |
|------|------|------|
| 401 | 401 | 未登录或 Token 无效 |
| 403 | 403 | 已登录但权限不足 |

**403 响应示例**

```json
{
  "code": 403,
  "message": "Forbidden: insufficient permissions",
  "data": null
}
```

## 前端落地说明

- `useAuthStore().can('task:run')`：判断当前用户是否有权限。
- `usePermission()`：组件内使用的 composable。
- 路由 `meta.permission`：页面级访问控制，无权限跳转 `/forbidden`。
- 按钮/菜单：`v-if="can(PERMISSIONS.xxx)"` 控制显隐。

## 后端实现建议（给 Manager 团队）

1. 用户表：`user_id, username, password_hash, name, email, role, status`
2. 角色权限：可先硬编码三张映射表（与本文档一致），后续再抽 `role_permissions` 表。
3. 登录成功：签发 JWT 或 session token=${ACCESS_TOKEN} `permissions[]`。
4. 中间件：解析 Token → 加载用户 → 注入 `permissions` → 路由守卫校验。
5. 禁用用户：`status=disabled` 时拒绝登录。
